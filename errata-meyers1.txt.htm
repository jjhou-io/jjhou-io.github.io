書籍內容更正，有兩種作法，一是在網際網路上做個專屬勘誤
網頁，讓大家上去看。這是比較即時的作法。而更理想更負責
的作法是：不但有勘誤網頁，並且在新刷中予以更正 — 如果
有新刷的話 :)
不過，理想與現實之間需要一點協調。書籍的製作是這樣的，
製版與印刷時，是以檯（8 或 16 頁）為單位。因此，每換一
頁，同檯的各頁統統要換過。這便造成印製成本的大量增加。
以前，我從不考慮成本，只要我認為書籍內容有修改必要，即
使只是某個字詞用得不甚理想，我都會請出版社更新。出版社
也都全力配合（這一點讓我非常感謝）。
慢慢地，我的行事不再這麼霹靂，我覺得我多少也要站在出版
社的立場想想。所以我打算，如果是關係到對錯正誤的根本性
問題，我便一定在新刷修正。如果是易判別的錯別字或排版誤
失或用詞不很恰當…等等，我便先在勘誤網頁上明載，但不求
一定於新刷中更正。直到收集來的這類誤失較為密集了，才一
併於下一刷修正。
哪些是新刷已修正的，哪些是暫請讀者自行動手更改的，我會
在勘誤網頁（網址見書封底）上清楚說明。
這是個便宜法門，請讀者見諒。
書籍應該在出版前就詳細檢查，以完美之姿出現。但是完美很
難達到。對於下列大大小小輕重不等的誤失，我謹向讀者說抱
歉。
本檔歡迎廣為流傳，謝謝。

■p142, L1（誤譯）
原文：不過，這段碼是否合法並非重點所在（譯註：畢竟只是一段虛擬碼），
      重點是它在必要情況下呼叫了operato new，用以建構 Derived 的
      「base class 成份」以及各個 data members。這個呼叫動作可能被
      默默安插到你的 constructors 內。
更正：不過，這段程式碼是否合法並非重點所在（譯註：畢竟只是一段虛擬碼），
      重點是它（必要時）呼叫了operator new、建構了 Derived 的
      「base class 成份」和各個 data members，而這段碼可能被
      默默安插到你的 construcors 內。
討論：new expression 的動作實際分為兩階段：
          (1) 呼叫 ::operator new 配置記憶體
          (2) 呼叫 constructor 將物件建構起始（設立初值）
      上述的 (1) 獨立於 (2) 之外。但本書 p141 下半頁的虛擬碼卻將 (1) 置於
      (2) 之內，這是不對的。
      因此，依我（侯捷）之見，本書 p141 下半頁的虛擬碼中，::operator new 的
      呼叫動作應該剔除於 constructor 之外。而上述的描述文字也應該修改為：
      不過，這段程式碼是否合法並非重點所在（譯註：畢竟只是一段虛擬碼），
      重點是它建構了 Derived 的「base class 成份」和各個 data members
      （甚至在繼承並帶有虛擬函式的情況下，還需妥善設定 vptr），而這些碼可能被
      編譯器默默安插到你的 construcors 內。

原文：
第一個理由是你可能想要「擒抱」訂製型記憶體管理系統。有許多理由你會這麼做，
最平常的一個理由就是，預設的 operator new 和 operator delete 對你而言使用了
太多記憶體或是速度太慢（或兩者兼具）。正是這樣，每一個你貢獻出來用以追蹤和

受到約束，所以 pool 作法可能是最好的一種策略。
更正：
第一個理由是你可能想要「擒抱」訂製型記憶體管理系統。有許多理由你會這麼做，
最平常的一個理由就是，預設的 operator new 和 operator delete 對你而言使用了
太多記憶體或是速度太慢（或兩者兼具）。正是這樣，每一個你貢獻出來用以追蹤和
釋放那些大塊記憶體的額外記憶體空間和額外的程式碼都筆直地達陣：如果不採用 pool
策略，你的軟體跑起來會比較慢而且耗用較多記憶體。對那些抱持「效率乃十分必要」
之態度的程式庫和應用軟體而言（並且由於 pool 的大小可受到適度約束），
pool 可能是最好的一種策略。
